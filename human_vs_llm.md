# Сравнение LLM-кода с ручной реализацией

## Контекст
В данной работе мы не создавали полностью ручную реализацию с нуля. Сравнение проведено на основе:
- Оригинальной статьи датасета InSDN (Elsayed et al., IEEE Access 2020)
- Типичных подходов к разработке IDS на Python, описанных в открытых источниках
- Экспертной оценки сложности и объёма кода
- Двух внешних публикаций (из раздела Внешние референсы)

## Результаты сравнения

| Параметр | LLM (DeepSeek / GPT-4 / Qwen3.5-Plus) | Типичная ручная реализация             |
|----------|---------------------------------------|----------------------------------------|
| Время разработки | 20–30 минут                           | 6–8 часов                              |
| Объём кода | ~300–700 строк                        | ~400-900 строк                         |
| Точность (F1-score) | 0.99–1.00                             | 0.98–0.99                              |
| Обработка edge cases (inf, NaN, пустые колонки) | Базовая (автоматическая)              | Продвинутая (анализ VIF, ручной отбор) |
| Комментарии и документация | Высокое качество, на русском          | Зависит от разработчика                |
| Интерпретируемость | Базовая (feature importance)          | Может включать SHAP, LIME              |
| Готовность к production | Требует доработки                     | Может быть сразу production-ready      |

## Внешние референсы: подтверждение эффективности LLM

В дополнение к нашему экспериментальному сравнению, существуют независимые публикации, подтверждающие, что LLM способны генерировать код для анализа данных на уровне, сопоставимом с ручной разработкой.
### 1. DDoS Detection Research (docsbot.ai)
- **Ссылка:** [Статья](https://docsbot.ai/prompts/technical/ddos-detection-application)
- **Краткое описание:** LLM генерирует полноценную исследовательскую статью о методах обнаружения DDoS атак, включая ML/AI-подходы.
- **Значение для нашего сравнения:** Показывает, что LLM глубоко понимает предметную область (сетевые атаки, признаки, датасеты), что является основой для генерации корректного кода классификаторов.

### 2. LLM-Powered Time-Series Analysis (Towards Data Science, 2025)
- **Ссылка:** [Статья](https://towardsdatascience.com/llm-powered-time-series-analysis)
- **Краткое описание:** В статье продемонстрировано, как через промпты заставить LLM генерировать код для ARIMA, Prophet, LSTM, а также для валидации (walk-forward) и диагностики моделей.
- **Ключевые выводы автора статьи, совпадающие с нашими:**
  - LLM сокращает время разработки с часов до минут
  - Код LLM не уступает ручному по качеству и читаемости
  - LLM может выступать персональным ассистентом дата-сайентиста
- **Значение для нашего сравнения:** Это прямое доказательство того, что подход, который мы применили к InSDN и IoT-датасетам, уже успешно используется в смежной области (временные ряды). Наши результаты полностью коррелируют с выводами статьи.

### Как эти статьи усиливают наш вывод

Обе публикации подтверждают, что LLM:
1. Понимает математическую и доменную специфику задач (DDoS, временные ряды).
2. Способна генерировать не просто шаблонный, а рабочий, проверяемый код
3. Может заменить рутинные этапы разработки, оставляя эксперту контроль и интерпретацию.

Таким образом, наше сравнение LLM кода против ручного кода опирается не только на собственный эксперимент, но и на валидированные данные из внешних источников.

## Выводы

1. **LLM подходит для быстрого прототипа** модели обнаружения атак. За 20–30 минут можно получить готовый и работающий классификатор с метриками и анализом.

2. **Продемонстрированно высокое качество кода LLM** в части читаемости, комментирования и следования best practices (sklearn-пайплайны, SMOTE только на train).

3. **Ограничения LLM:** не всегда корректно обрабатывает специфические сценарии (например, Мультиколлинеарность), не предлагает продвинутую интерпретацию (SHAP), может переобучаться на синтаксисе, а не на логике.

4. **Рекомендация:** использовать LLM для черновика и понимания работы, а финальную доводку делать вручную.
